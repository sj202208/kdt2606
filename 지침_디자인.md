## 문서의 목적

이 파일은 AI 에이전트(Claude, Cursor, Stitch, Gemini CLI 등)에게 전달하는 **재사용 가능한 지시문**이다. 사용자는 이 md를 채팅에 첨부한 뒤, **같은 메시지 본문에 원하는 시각 정체성을 텍스트로 묘사**한다 (예: 무드, 도메인, 타겟 사용자, 브랜드 레퍼런스, 명시적인 색상·폰트 지정, 회피해야 할 패턴 등).

AI는 측정이 아닌 **합성 절차**로 묘사를 구체적인 토큰 값으로 번역하고, Google이 공개한 [DESIGN.md spec](https://github.com/google-labs-code/design.md)을 따르는 단일 파일을 반환한다. 결과물은 프로젝트 루트의 `DESIGN.md`로 그대로 복사·붙여넣기 가능하다.

URL 입력으로 라이브 사이트를 분석하려면 별도의 URL 버전 프롬프트(`지침_디자인문서만들기_URL.md`)를 사용한다.

> **본 지침의 위치**: 마스터 지침(`지침_통합.md` v3.0)의 부속 모듈. DESIGN.md 입력 모드 두 가지(URL / 텍스트 묘사) 중 텍스트 묘사 모드를 담당한다.

## 사용 방법

1. 이 파일을 AI 채팅에 첨부한다.
2. 같은 메시지에 텍스트 묘사를 입력한다.
   - **권장 길이**: 최소 1–2 문단 (묘사가 한두 단어로만 끝나면 AI가 한 번 되물음).
   - **포함하면 좋은 정보**: 무드·톤, 도메인·타겟, 명시적 색상·폰트, 브랜드/레퍼런스, 회피할 패턴, 다크모드 여부, 접근성 요구 수준.
   - PRD.md 부속 D(레퍼런스 / 분위기)를 그대로 복사해 입력해도 된다.
3. 받은 단일 fenced Markdown 블록을 프로젝트 루트의 `DESIGN.md`로 저장한다.
4. (옵션) lint CLI로 검증: `npx @google/design.md lint DESIGN.md`. 초보자는 생략 가능.
5. AI가 본문에 명시한 "해석한 선택" 항목을 검토하고, 어긋나는 부분은 다음 턴에서 교정한다.
6. 이 DESIGN.md를 Stitch에 입력해 사이트 UI를 export 받는다. Cursor에는 export 결과물(HTMLs + DESIGN2.md)이 입력된다.

# Instructions to the AI

You are a DESIGN.md author. This file is the instruction set; the actual input — a prose description of the desired visual identity — arrives in the chat message that accompanies this attachment.

**Minimum input check**: if the description is shorter than ~2 sentences or contains only a single attribute (e.g., "make it modern", "blue"), ask the user once for more detail before generating. Specifically prompt for: mood / tone, domain / audience, any required colors or fonts, brand references, and anti-patterns. Do not fabricate a system from a one-word brief.

**Goal**: produce a single, copy-pastable DESIGN.md file that an AI design agent (Stitch, Claude Code, Cursor, Gemini CLI, etc.) can consume to reproduce the intended visual identity.

**Reference material**

- Overview: https://stitch.withgoogle.com/docs/design-md/overview/
- Format: https://stitch.withgoogle.com/docs/design-md/format/
- Spec: https://github.com/google-labs-code/design.md
- Example — atmospheric-glass: https://github.com/google-labs-code/design.md/blob/main/examples/atmospheric-glass/DESIGN.md
- Example — paws-and-paths: https://github.com/google-labs-code/design.md/blob/main/examples/paws-and-paths/DESIGN.md

## Synthesis procedure

1. **Read the description and extract:**
   - **Mood and tone** (editorial, playful, brutalist, glassy, minimal, maximalist, etc.).
   - **Domain and audience** (consumer / enterprise / kids / luxury / B2B SaaS / fintech / health / etc.).
   - **Explicit constraints**: stated colors, fonts, brand references, or anti-patterns ("no gradients", "not corporate", "must work in dark mode").
   - **Implicit constraints**: accessibility expectations, information density, formality, motion appetite.
2. **Translate references into tokens, do not copy them.** When the description names a brand, a site, a design movement, or a physical material as a stylistic anchor, render it as concrete token values (hex, rem, px) — never reproduce a third party's identity verbatim.
3. **Synthesize a coherent, internally consistent system** rather than over-fitting to literal wording. Where the description is silent, fill gaps with sensible brand-consistent defaults that the description does not contradict. Prefer a small, semantically named token set; avoid inflating the system with values the brief never asked for.
4. **Make interpretation visible.** In the Markdown body, briefly state the interpretive choices made — for example: "Interpreted *warm and editorial* as serif headings (Source Serif), warm limestone surface (#F7F5F2), and a single terracotta accent (#B8422E) reserved for primary action." This lets the user correct the system in a follow-up turn.

---

## Output requirements

- Begin the file with **YAML frontmatter** containing all structured design tokens (colors, typography, spacing, rounded, elevation / shadows, motion, components, etc.), conforming to the DESIGN.md spec.
- Use **spec-valid token types**: `"#RRGGBB"` sRGB hex for colors, number+unit (`px` / `em` / `rem`) for dimensions, `"{path.to.token}"` for token references inside components.
- Place `##` sections in the **canonical order** defined by the spec: Overview, Colors, Typography, Layout, Elevation & Depth, Shapes, Components, Do's and Don'ts. Omit any section that does not apply, but never reorder.
- Follow the frontmatter with **free-form Markdown** that captures look & feel and the design intent that token values alone cannot convey: mood, hierarchy, restraint vs. expressiveness, when to use each color / component, accessibility posture. Include the "interpretive choices" note from synthesis step 4.
- The file must be **entirely self-contained** — do not reference the original chat message, external assets, codebase paths, or any variable not defined inside the file itself.
- Prefer a **small, semantically named token set** over an exhaustive raw dump.

---

## Validation pass

- Re-read the original description and verify the resulting system "feels" consistent with it. List any deliberate deviations in the Markdown body so the user can review them.
- Mentally apply the official lint rules: `broken-ref`, `missing-primary`, `contrast-ratio`, `orphaned-tokens`, `missing-typography`, `section-order`. Every `{token.ref}` must resolve. Component `backgroundColor` / `textColor` pairs must meet WCAG AA (≥ 4.5:1 for body text). Sections must be in canonical order.
- Revise until both the YAML tokens and the prose faithfully express the intent of the description.

---

## Deliverable

A single fenced Markdown block containing the complete DESIGN.md, ready to copy and paste into the project root. No preamble, no postscript, no explanation outside the fence — just the file.
