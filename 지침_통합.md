> **이 문서의 목적**
> 6개 마크다운 문서(PRD / DESIGN / RULES / BACKLOG / RUN-BACKLOG / SKILL)의 **역할·필수 항목·연결 관계**만 정의한다.
> 개별 프로젝트의 실제 내용은 본 지침과 4개 부속 지침을 사용해 GPT/Cursor가 산출한다.
> 본 지침에는 특정 프로젝트(예: 영어 문제 풀기 사이트) 색채를 포함하지 않는다.

---

## 1. 큰 그림 — "왜 6개 문서를 쓰나?"

AI는 한 번에 모든 것을 기억하지 못한다. 사람의 머릿속 생각을 6개의 짧은 마크다운으로 나누면, 각 문서는 **한 가지 질문에만 답한다**.

| 문서 | 답하는 질문 | 비유 | 작성 주체 |
|---|---|---|---|
| **PRD.md** | 무엇을, 왜 만드는가? | 설계도의 목적 | 사람 |
| **DESIGN.md** | 어떻게 보이게 할까? | 인테리어 컨셉 | 사람 + Stitch |
| **RULES.md** | 뭘 하면 안 되고, 뭘 꼭 지켜야 하나? | 작업 안전수칙 | GPT (지침 기반) |
| **BACKLOG.md** | 뭘 먼저 만들고, 끝난 걸 어떻게 알까? | 할 일 목록 + 검수표 | GPT (지침 기반) |
| **RUN-BACKLOG.md** | 어떤 순서로 돌릴까? | 작업 지시서 | GPT (지침 기반) |
| **SKILL.md** | 반복되는 일은 어떻게 자동화하나? | 도구 매뉴얼 | GPT (지침 기반) |

> **암기 팁**: "**왜(PRD) → 어떻게 보일지(DESIGN) → 안전수칙(RULES) → 할 일(BACKLOG) → 순서(RUN-BACKLOG) → 도구(SKILL)**"

---

## 2. 작성 주체와 책임 분리

이 메소돌로지의 핵심은 **사람이 쓰는 영역**과 **GPT가 만드는 영역**의 분리다.

```
[사람이 직접 쓴다]
  └─ PRD.md         (기획 PRD 가이드 참조)
  └─ DESIGN.md 입력  (URL 또는 텍스트 묘사)

           ↓

[Stitch가 만든다]
  └─ DESIGN.md 산출물 (HTMLs + DESIGN2.md)

           ↓

[GPT가 만든다 — 개발문서 지침이 시스템 프롬프트]
  └─ RULES.md
  └─ BACKLOG.md
  └─ RUN-BACKLOG.md
  └─ SKILL.md

           ↓

[Cursor/Antigravity가 빌드한다]
  └─ 사이트
```

---

## 3. 전체 워크플로우 (6단계)

```
0. 사용자
   ├── 기획지침   → ① PRD.md (사람이 작성)
   └── 디자인지침 → ② DESIGN.md (사람이 초안)
                         ↓
                  ③ STITCH (export)
                         ↓
                  ④ HTMLs/IMAGEs + DESIGN2.md (확정)
                         ↓
   개발지침 ┄┄┄┄┄→ ⑤ Cursor / Antigravity
                  ├── PRD.md / DESIGN2.md / HTMLs (입력)
                  ├── RULES.md (GPT 산출)
                  ├── BACKLOG.md (GPT 산출)
                  ├── RUN-BACKLOG.md (GPT 산출)
                  └── SKILL.md (GPT 산출)
                         ↓
                       build
                         ↓
                  ⑥ 사이트
```

> **다이어그램 화살표 규칙**: 화살표가 Cursor 박스 경계에 닿으면 그 문서는 Cursor 안으로 들어간다는 뜻.

## 4. 부속 지침 사용법

본 마스터 지침은 4개 부속 지침과 함께 사용한다. 각 지침의 진입 시점은 다음과 같다.

| 단계 | 사용 지침 | 산출물 |
|---|---|---|
| ① PRD 작성 | `지침_기획_PRD_문서만들기.md` | PRD.md (사람이 직접 작성) |
| ② DESIGN 입력 (URL) | `지침_디자인문서만들기_URL.md` | DESIGN.md (Stitch 입력용) |
| ② DESIGN 입력 (텍스트) | `지침_디자인문서만들기_설명.md` | DESIGN.md (Stitch 입력용) |
| ③④ Stitch export | (도구 자체) | HTMLs + DESIGN2.md |
| ⑤ 개발 문서 자동 생성 | `지침_개발문서만들기.md` | RULES / BACKLOG / RUN-BACKLOG / SKILL |

**중요**: ②의 두 지침(URL/설명)은 Stitch 진입 전 분기다. Stitch 이후 Cursor에 입력되는 것은 **단일 DESIGN2.md** 하나다.

---

## 5. 사람이 쓰는 두 문서의 양식

본 지침에는 사람이 직접 쓰는 두 문서(PRD / DESIGN)의 빈 양식만 포함한다. RULES·BACKLOG·RUN-BACKLOG·SKILL은 GPT가 `지침_개발문서만들기.md`를 사용해 자동 생성하므로, 본 지침에서는 양식을 중복 보유하지 않는다.

---

### 5.1 PRD.md — 빈 양식

상세 작성법은 `지침_기획_PRD_문서만들기.md`를 참조한다. 본 양식의 6개 정식 섹션은 후행 GPT가 받아들일 수 있는 표준이다.

```markdown
# PRD — [프로젝트명]

## 1. 서비스 정의
[누가 / 어떤 문제를 / 어떻게 해결하는지 한 문장]

## 2. 타겟 유저
- 사용자: [누가]
- 상황: [언제, 어디서, 어떤 기기로]
- 디지털 친숙도: [낮음/중간/높음]

## 3. 사용자 시나리오
### 성공
1. [단계 1]
2. [단계 2]
...

### 예외
- [예외 상황 1과 처리 방식]
- [예외 상황 2와 처리 방식]

## 4. 프로젝트 범위
### 반드시 되어야 함 (Scope In)
- [필수 기능 1]
- [필수 기능 2]

### 절대 안 됨 (Scope Out)
- [금지 동작 1]
- [금지 동작 2]

## 5. 화면 목록
1. [화면 이름] — [화면 역할]
2. [화면 이름] — [화면 역할]

## 6. 완료 기준
- [ ] [완료 조건 1: "~하면 OK"]
- [ ] [완료 조건 2]
- [ ] [완료 조건 3]

---

## (부속 A) 운영 제약
[예산 / 유지보수 / 모바일·PC 우선 / 외부 연동 등]

## (부속 B) 데이터 목록
| 데이터명 | 필수 | 수정 권한 | 민감도 | 저장 |
|---|---|---|---|---|
| [예: 사용자 이름] | ✅ | 본인 | 일반 | DB |

## (부속 C) 예시 입력 / 출력
- 입력: [예시 1]
- 출력: [예시 1]

## (부속 D) 레퍼런스 / 분위기
- 참고 서비스: [URL 또는 캡처] — [참고할 점 1줄]
- 원하는 느낌: [형용사 2~3개]
- 피하고 싶은 스타일: [형용사 2~3개]
```

**필수/부속 구분**
- 정식 섹션 1~6: 모두 필수. 비어 있으면 GPT가 RULES·BACKLOG 생성 시 사용자에게 질문한다.
- 부속 A~D: 권장. 비어 있어도 GPT는 동작하며, 필요 시 합리적 기본값으로 채운다.

---

### 5.2 DESIGN.md — 빈 양식 (구조 이해용)

> **중요**: 아래 빈 양식은 **구조 이해용 참고**다. 실제 DESIGN.md는 직접 채우지 않고 `지침_디자인문서만들기_URL.md` 또는 `지침_디자인문서만들기_설명.md`를 사용해 **Google Stitch spec(alpha) 풀 버전으로 자동 생성**한다. 두 지침이 산출하는 결과는 본 양식보다 풍부하며, 그 결과를 그대로 Stitch에 입력해 사이트 UI를 export 받는다.

```markdown
---
# YAML frontmatter (Stitch spec 토큰 정의)
colors:
  primary: "#______"
  background: "#______"
  text:
    onPrimary: "#______"
    onBackground: "#______"
typography:
  body: { fontFamily: "______", fontSize: "1rem" }
  heading: { fontFamily: "______", fontSize: "2rem", fontWeight: 600 }
spacing: { sm: "8px", md: "16px", lg: "24px" }
rounded: { sm: "6px", md: "12px" }
components:
  buttonPrimary:
    backgroundColor: "{colors.primary}"
    textColor: "{colors.text.onPrimary}"
    borderRadius: "{rounded.md}"
---

## Overview
[브랜드 분위기, 톤, 절제·표현 정도]

## Colors
[각 토큰의 사용처와 의미]

## Typography
[폰트 선택의 근거, 위계]

## Components
[컴포넌트별 토큰 매핑과 변형]
```

**필수 섹션 (Stitch spec 표준 순서)**: Overview / Colors / Typography / Layout / Elevation & Depth / Shapes / Components / Do's and Don'ts. 적용되지 않는 섹션은 omit 가능하지만 순서는 유지한다.

**lint 검증**: 옵션. 초보자는 생략 가능. 중급은 AI에게 mental lint 요청. 고급은 `npx @google/design.md lint DESIGN.md` 사용 (Node.js 필요).

## 6. GPT가 만드는 4문서의 산출 책임

본 지침은 양식을 보유하지 않는다. 다음 매핑으로 `지침_개발문서만들기.md`(시스템 프롬프트)가 산출한다.

| 산출물 | 입력 | GPT가 채우는 것 |
|---|---|---|
| RULES.md | PRD + DESIGN(있으면) | 행동 규칙(고정) + 프로젝트 규칙(빈 꼭지에 사용자 답변·DESIGN 게이트 자동 추가) |
| BACKLOG.md | PRD | US + AC (URL/텍스트/요소 상태 명시) |
| RUN-BACKLOG.md | (고정 템플릿) | 4단계 워크플로우 |
| SKILL.md | BACKLOG의 AC 형식 | ac-to-test 변환 규칙 |

## 8. 환경 표준 (Windows CMD 고정)

본 지침은 Windows CMD를 표준 환경으로 정의한다. Mac/Linux는 별도 부록에서 다룬다.

- 가상환경 실행: `.venv\Scripts\python` 직접 경로
- `activate.bat` / `source .venv/bin/activate` 사용 금지
- 셸 명령은 `cmd /c`로 감싸서 실행
- 자동 세팅: `setup.bat` (개발문서 지침에서 자동 생성)

## 9. AI에게 시키는 한 줄 명령

모든 문서가 준비되면, Cursor/Antigravity에서 다음 한 줄로 빌드를 시작한다.

```
@Files RUN-BACKLOG를 보고 순서대로 실행해줘.
```

## 10. 강의용 자가 점검 체크리스트

본 지침에 따라 산출된 6개 문서가 정합한지 확인한다.

- [ ] PRD의 정식 섹션 6개가 모두 채워져 있는가
- [ ] DESIGN.md가 Stitch spec의 섹션 순서를 따르는가
- [ ] RULES.md에 행동 규칙 3개가 고정 포함되어 있는가
- [ ] RULES.md의 프로젝트 규칙 섹션 4개 꼭지가 모두 존재하는가 (내용은 비어 있어도 됨)
- [ ] BACKLOG의 모든 AC가 URL / 정확한 텍스트 / 요소 상태 중 하나로 적혀 있는가
- [ ] RUN-BACKLOG가 Phase 0~4 구조를 갖는가
- [ ] SKILL.md에 YAML frontmatter와 Constraints가 있는가
- [ ] 모든 명령이 `.venv\Scripts\python` 직접 경로인가
