## ✍️ 기획자가 직접 작성하는 영역

PRD.md는 **6개의 정식 섹션**과 **4개의 부속 섹션**으로 구성된다. 정식 섹션은 모두 필수, 부속 섹션은 권장이다.

---

### 정식 섹션 (필수 6개)

이 6개가 후행 GPT가 RULES·BACKLOG를 산출하는 입력이다.

#### 1. 서비스 한 줄 정의 ✅ 필수
> 누가 / 어떤 문제를 / 어떻게 해결하는지 한 문장으로

#### 2. 타겟 사용자 ✅ 필수
> 누가 쓰는지 / 어떤 상황과 기기에서 쓰는지 / 디지털 친숙도

#### 3. 핵심 사용 시나리오 ✅ 필수
> 성공 흐름 3~5개 + 예외 상황 1~2개

#### 4. 프로젝트 범위 ✅ 필수
> Scope In (반드시 되어야 함) + Scope Out (절대 안 됨)

#### 5. 주요 화면 목록 ✅ 필수
> 화면 이름 + 각 화면에서 무엇을 하는가
> 복잡한 흐름은 손그림 또는 캡처 첨부

#### 6. 완료 기준 ✅ 필수
> "어디까지 되면 MVP 완성인가" — AI용 채점표
> 형식: "~하면 OK" 단순 명제

---

### 부속 섹션 (권장 4개)

비어 있어도 GPT가 동작하지만, 적어두면 결과 품질이 올라간다.

#### A. 운영 제약 🔵 권장
> 예산 · 유지보수 난이도 · 모바일/PC 우선 · 로그인 여부 · 외부 연동 여부

#### B. 데이터 목록 🔵 권장
> 다루는 정보 목록 + 필수 여부 + 수정 권한 + 민감도 + 저장 여부 (표 형식)

#### C. 예시 입력 / 출력 🔵 권장
> 실제 사용 예시 1~2개 (AI는 예시에 강하다)

#### D. 레퍼런스 / 분위기 🔵 선택
> 참고 서비스 링크 또는 캡처 + 어떤 점을 참고할지 한 줄 + 원하는 느낌 + 피하고 싶은 스타일

```
- 참고 서비스: [URL] — 문제가 하나씩 단순하게 나열되는 구조
- 원하는 느낌: 흰 배경, 집중하기 좋은 레이아웃, 버튼이 크고 명확함
- 피하고 싶은 스타일: 화려한 색상, 광고 같은 배너, 복잡한 메뉴
```

> **연결 관계**: 부속 D(레퍼런스 / 분위기)는 DESIGN.md 작성 시 입력으로 그대로 활용된다. URL이 있으면 `지침_디자인문서만들기_URL.md`로, 텍스트만 있으면 `지침_디자인문서만들기_설명.md`로 진입한다.

## 🤖 AI가 초안을 제안하고 사람이 확정하는 영역

> AI가 잘못된 제안을 할 수 있다. AI는 초안 제안자, 최종 확정은 기획자 또는 개발자가 한다.

| # | 항목 | 필수 여부 | AI 요청 방법 |
|---|---|---|---|
| 1 | 데이터 구조 초안 | ✅ 필수 | 데이터 목록을 주고 → 표 형태로 정리 후 → JSON 또는 DB 스키마 초안 요청 |
| 2 | 기술 스택 선택 | ✅ 필수 | 운영 제약을 주고 → 2~3안 비교 + 장단점 + MVP 권장안 요청 |
| 3 | 품질·테스트 기준 | ✅ 필수 | 완료 기준을 주고 → "어디까지 확인되면 완성인가" 검증 항목 목록 요청 |
| 4 | 구현 순서 제안 | 🔵 권장 | 기능 목록을 주고 → 개발 우선순위 순서 요청 (화면 3개 이상이면 사실상 필수) |
| 5 | 구현 원칙 (Coding Rules) | 🔵 선택 | 운영 제약을 주고 → 비개발자 언어로 된 품질 기준 요청. 협업·재사용 프로젝트면 권장 |

> **주의**: 위 5개는 **PRD 작성 시 AI에게 도움받는 영역**이다. RULES.md / BACKLOG.md 자체는 PRD가 완성된 뒤 `지침_개발문서만들기.md`(개발문서 GPT v9.0)가 산출한다.

## 작성 후 자가 점검

PRD.md 작성을 끝낸 뒤 다음 6가지를 확인한다. 통과하면 후행 단계가 막힘없이 진행된다.

- [ ] 정식 섹션 6개가 모두 채워져 있는가
- [ ] 1번(서비스 정의)이 한 문장 이내로 표현되었는가
- [ ] 3번(시나리오)에 성공·예외가 모두 있는가
- [ ] 4번(범위)의 Scope In과 Scope Out이 명시적으로 분리되어 있는가
- [ ] 6번(완료 기준)이 "~하면 OK" 형태로 6~10개 적혀 있는가
- [ ] 부속 D(레퍼런스 / 분위기)가 비어 있다면, DESIGN.md 작성 단계에서 텍스트로 묘사할 준비가 되었는가

## 💡 핵심 원칙

> **기획자는 목표·맥락·제약·성공 기준을 쓴다.**
> **AI는 그 위에 기술 초안과 선택지를 제안하고, 사람이 최종 확정한다.**

- 기술 스택을 기획자가 직접 확정하지 않아도 된다
- 단, AI가 올바른 방향으로 제안하려면 **운영 제약과 우선순위를 먼저 제공**해야 한다
- 와이어프레임은 필수가 아니지만, 예약·결제·대시보드처럼 상태 변화가 많은 서비스는 간단한 스케치라도 있으면 결과물 품질이 올라간다
